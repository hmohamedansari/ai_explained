## Module Details

### 6.1 — Why Evaluation Is Hard
**Personas:** All Devs
**Key concepts:** The Three Gulfs:
- **Comprehension gulf:** You cannot review AI output at production scale manually.
- **Specification gulf:** Prompts never fully capture what "good" means.
- **Generalisation gulf:** Models fail inconsistently on edge cases — passing a test suite doesn't mean the system is reliable.

**Production gotcha:** Agents built without an eval harness are unsteerable. You cannot improve what you cannot measure. Read this before building agents.
**Unknown Unknown:** Listed in `unknown-unknowns.md`.
**Note:** This module appears early in Paths C and D for a reason. Evaluation is a design input, not a post-hoc check.

---

### 6.2 — Code-Based Evals
**Personas:** Jr Dev →
**Key concepts:** Deterministic checks for deterministic tasks — cheap, CI/CD-friendly, and always the eval you should start with.
**Note:** Layer 2 must include a working eval harness for a simple task (e.g., testing that structured output always contains required fields). Make it runnable in a CI pipeline.

---

### 6.3 — LLM-as-Judge
**Personas:** Sr Dev
**Key concepts:** When to use LLM-as-judge, why binary PASS/FAIL outperforms Likert scales, hand-labelling with written critiques (not just labels), judge drift over time.
**Production gotcha:** LLM judges drift over time as the underlying model is updated. Version-pin your judge model and re-validate your eval suite when you upgrade it.
**Unknown Unknown:** Listed in `unknown-unknowns.md`.

---

### 6.4 — Error Analysis Workflow
**Personas:** Sr Dev
**Key concepts:** Open coding → axial coding → quantitative prioritisation (adapted from grounded theory). This is how production teams actually debug failing systems.
**Production gotcha:** Most teams skip error analysis and jump straight to prompting changes. The result is thrashing — improving one failure class while regressing another. Error analysis finds the highest-leverage fix.
**Unknown Unknown:** Listed in `unknown-unknowns.md`.

---

### 6.5 — RAGAS & RAG Evaluation
**Personas:** Jr Dev →
**Key concepts:** Faithfulness (does the answer match the retrieved context?), context recall (did retrieval find the right chunks?), answer relevancy.
**Note:** Layer 2 must run a real RAGAS evaluation against a small document corpus. Don't just explain the metrics — show what a failing score looks like and what it tells you to fix.

---

### 6.6 — Synthetic Data for Eval
**Personas:** Sr Dev
**Key concepts:** Bootstrapping evaluation before real production traffic exists. Using the LLM to generate test cases, then validating those test cases.
**Production gotcha:** Synthetic data generated by the same model you're evaluating can produce circular validation. Use a different, stronger model for generation, or generate adversarial cases specifically designed to fail.

---

### 6.7 — CI/CD-Integrated Evals
**Personas:** Sr Dev / SRE
**Key concepts:** Eval as a test gate, regression detection, model upgrade validation.
**Note:** Layer 2 must include a working GitHub Actions (or equivalent) configuration that runs an eval suite on every PR and blocks merge on regression.

---

### 6.8 — Observability Stack
**Personas:** All Devs
**Key concepts:** Trace-level vs. log-level visibility. Key tools:
- **Langfuse** — open-source tracing, evals, and prompt management
- **LangSmith** — advanced debugging for LangChain/LangGraph
- **Arize Phoenix** — embedding visualisation and tracing
- **Helicone** — proxy-based observability for costs and latency

**Note:** Cover the difference between trace-level observability (what happened inside a single LLM call) and system-level observability (what happened across an entire agent run). Both are needed.

---

### 6.9 — Cost Management
**Personas:** SRE / Leader
**Key concepts:** Token counting, prompt caching, cost ceilings, billing models for multi-agent systems.
**Note:** Layer 1 is for Leaders (cost dashboards, budgeting). Layer 2 is for SREs (implementing caching and cost ceilings in LiteLLM). Cross-reference 5.8 (LLM Gateway Patterns).

---

### 6.10 — Evaluating the Evaluator
**Personas:** Sr Dev
**Key concepts:** Production monitoring that validates your evals themselves remain trustworthy — the meta-loop most teams skip.
**Production gotcha:** Your eval suite can silently drift out of alignment with what "good" means as your product evolves. Without a process for evaluating the evaluator, you are flying blind.
**Unknown Unknown:** Listed in `unknown-unknowns.md`.
